## 1. Problem Statement

SK ID Solutions is a trust service provider at the center of Estonia's digital identity infrastructure. The EU Digital Identity Wallet (EUDI Wallet) under eIDAS 2.0 is the single most important regulatory and product development topic affecting SK's future.

The EUDI Wallet landscape is fragmented across dozens of information sources — EU institutional pages, technical standards bodies, national implementation updates, GitHub repositories, industry blogs, and expert commentary. Staying current requires manually checking these sources, filtering noise from signal, and synthesizing developments into actionable intelligence.

**No one at SK has a systematic, automated way to:**
1. Monitor all relevant EUDI Wallet information sources
2. Get curated, deduplicated, AI-filtered updates on a regular cadence
3. Maintain a living strategic document that evolves as the topic evolves
4. Share this intelligence effortlessly with the team

---

## 2. Solution Overview

An automated intelligence pipeline with two systems working together:

1. **Render worker** (autonomous, AI-free) scrapes a configurable list of sources **twice per week**, validates and deduplicates articles, stores them as raw `pending` entries in Neon Postgres.
2. **Claude Code loop** (owner's subscription, zero extra cost) processes pending articles — scores relevance, generates summaries, updates a living Google Doc, and sends a newsletter email twice per month.

**Three outputs:**
- **Public news feed dashboard** — curated articles, updated after every scrape cycle (twice/week)
- **Email newsletter** — digest sent **twice per month** to a defined list of colleagues
- **Living strategy document** (Google Doc) — AI-maintained cumulative knowledge, shared with SK team

Scraping runs fully autonomously on Render. AI work uses the owner's existing Claude Code subscription — zero additional API costs. If the Claude Code loop hasn't run, raw articles queue safely in Neon until the next run.

### 5.2 Scraping Pipeline (Render — AI-Free)

| ID | Requirement | Priority |
|----|-------------|----------|
| SCR-01 | Twice-weekly automated scrape of all active sources | Must |
| SCR-02 | Runs on Render as a background worker — fully autonomous, no local machine dependency | Must |
| SCR-03 | Two parser modes: `rss` (rss-parser) and `css` (Cheerio with stored selectors). Add more modes later if sources require it. | Must |
| SCR-04 | Extracts: title, URL, publication date, author, source name, full text or excerpt | Must |
| SCR-05 | Graceful failure — one source failing doesn't block the rest | Must |
| SCR-06 | Scrape run logging — per-source success/failure/article count in `scrape_runs` table | Must |
| SCR-07 | **Deduplication** — URL hash (unique constraint) + content fingerprint (title + first 200 chars). Same article never stored twice. | Must |
| SCR-08 | New articles stored with `status = "pending"` — awaiting AI processing | Must |
| SCR-09 | Basic validation: articles exist, titles non-empty, URLs valid, dates not future-dated | Must |

**LLM-powered source onboarding** *(pattern from [ai-rss-feeds](https://github.com/leontloveless/ai-rss-feeds)):*

When adding a non-RSS source via admin, the system makes a single Claude API call (pay-per-use, ~$0.01 per source) to analyze the page HTML and generate CSS selectors. One-time per source, config stored in `sources.config` and reused every cycle. This is the only place the project uses the Claude API directly — all recurring AI work (article curation, living doc, newsletter) runs on the owner's Claude Code subscription.

**When a source breaks** (0 articles but page is reachable): source shows as "broken" in admin. Owner manually re-runs selector analysis from the admin interface. No automated self-healing in V1 — sites rarely redesign, and when they do, manual inspection is usually needed anyway.

**What we explicitly skip for V1:**
- LinkedIn scraping (walled garden)
- PDF extraction (deferred)
- Headless browser rendering (Cheerio only — lightweight, free-tier compatible)
- `json` parser mode (for SPAs) and `github-releases` mode — add when a source needs them

### 5.3 AI Curation (Claude Code Loop — Owner's Subscription)

All AI work runs inside a Claude Code loop registered in the Loop Control Center. Uses the owner's existing subscription — zero additional API costs.

| ID | Requirement | Priority |
|----|-------------|----------|
| AI-01 | Read all `pending` articles from Neon | Must |
| AI-02 | Score each article for EUDI Wallet relevance (1-10) | Must |
| AI-03 | Articles below threshold (default: 6) marked as `rejected` with reason | Must |
| AI-04 | Accepted articles get: 2-3 sentence summary + category tags | Must |
| AI-05 | Update article status in Neon → `accepted` or `rejected` | Must |
| AI-06 | If 0 pending articles → log "nothing to process", done | Must |
| AI-07 | Graceful backlog handling — if loop hasn't run for a while, process all accumulated pending articles in one batch | Must |

**Category taxonomy (initial, refinable):**
`regulation` · `technical-standards` · `national-implementation` · `industry` · `security-privacy` · `interoperability` · `market-analysis`

### 5.7 Living Strategy Document (Google Doc)

A single Google Doc that represents the team's cumulative, structured knowledge about the EUDI Wallet landscape. Not a feed — a **living reference document** that gets smarter with every cycle.

| ID | Requirement | Priority |
|----|-------------|----------|
| DOC-01 | A single Google Doc owned by Egert, shared with SK team | Must |
| DOC-02 | Document structure is **owner-defined** — Egert specifies the topics and sections | Must |
| DOC-03 | Updated **twice per month** (aligned with newsletter cadence, not every scrape). Batches ~2 weeks of accepted articles into one coherent doc update. | Must |
| DOC-04 | Written via Google Docs API using existing `gws docs` tooling | Must |
| DOC-05 | Versioned via Google Docs native version history | Must |

**How updates work:**

After processing pending articles, the Claude Code loop:
1. Reads the current Google Doc content
2. Reads newly accepted articles from this cycle
3. For each section: determines if new articles contain relevant new/changed/contradicting information
4. Updates only affected sections — preserves everything else untouched
5. Appends a changelog entry: "Updated [date]: [what changed and why, with source citations]"

**Core rules:**
- **Incremental updates only** — never rewrite sections with no new information
- **New facts** → add to relevant section, cite source
- **Updated facts** → replace old fact, note what changed: "(previously X, updated per [source, date])"
- **Contradictions** → present both versions with citations
- **Cumulative info** (country lists, timelines) → append, don't replace
- **No new info for a section** → leave it completely untouched

**Confirmed section (1 of ~3 key topics):**
- **National Implementations — Baltic focus** (Estonia, Latvia, Lithuania): Ongoing monitoring of each Baltic state's EUDI Wallet implementation status, pilot progress, institutional decisions, and timelines. Core strategic interest given SK's market position.

Remaining key topics TBD — dependent on source research and incoming data feeds.

### 7.4 Pipeline Flow

**PHASE A — Render (autonomous, twice-weekly):**
1. Load active sources from Neon
2. For each source: fetch → extract articles → validate → dedup
3. Write new articles to Neon as `pending`
4. Log scrape run stats

**PHASE B — Claude Code Loop (owner's subscription, twice-weekly):**
1. Read all `pending` articles from Neon
2. Score relevance, generate summaries, assign categories
3. Update statuses → `accepted` / `rejected`
4. Update living Google Doc with new knowledge
5. Report to Loop Control Center

**PHASE C — Newsletter (twice per month):**
1. Collect all accepted articles since last newsletter
2. Rank by relevance, generate digest
3. Send via Resend to hardcoded email list

Phase A runs autonomously on Render. Phase B runs when the Claude Code loop fires (ideally shortly after each scrape). Phase C triggers twice per month (every other loop run, or on a separate schedule). If any phase is delayed, articles queue safely.

## 10. Productization Mindset

This project is built for SK as the first user, but **not as a one-off internal tool.** Every decision assumes this could become a standalone product.

| Principle | How it shapes the build |
|-----------|------------------------|
| **Domain-agnostic core** | Scraping engine, AI curation, admin, and dashboard are not hardcoded to "EUDI Wallet." Swap sources and prompts → tracker for any regulatory domain. |
| **Admin as product interface** | Not a developer backdoor. Intuitive enough for a non-technical PM at another company. |
| **Self-service source onboarding** | "Paste URL → AI analyzes → preview → activate" is the killer productization feature. |
| **Branding separation** | Dashboard name, colors, copy not hardcoded to SK. Configurable. |
| **Clean data layer** | Neon schema supports future mobile apps, API access, integrations. |

**What this does NOT mean:** No multi-tenancy now. No billing. No over-engineering. Just don't paint into corners.

**Future path:** V1 (single tenant, SK) → V2 (multi-tenant, invite-based) → V3 (SaaS).

## 11. Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Source sites change structure | Shows as "broken" in admin. Owner re-runs analysis manually. |
| Claude Code loop expires (72h) | Articles queue as `pending`. Owner `/sync-loops` on next session. No data loss. |
| Owner doesn't open Claude Code for >2 weeks | Backlog grows but is processed in one batch on next run. Dashboard shows "awaiting review" count. |
| Render free tier cold start | node-cron runs inside the service. Add keep-alive ping if needed. |
| Low team adoption | Start with owner as sole user. Expand after proving value. |
| Two-dashboard complexity | Linked dashboards (Option A). Future: embed LCC widget in EUDI Admin. |
